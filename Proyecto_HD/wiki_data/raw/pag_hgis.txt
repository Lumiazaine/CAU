===== Formas de acceso a BMC Helix =====

</WRAP> A continuación se detallan los métodos de autenticación disponibles para acceder a la plataforma BMC Helix:

**1. Acceso mediante Usuario/Contraseña de TEMIS**

  * Se basa en las credenciales gestionadas TEMIS.
  * El acceso se realiza a través de **"Usuario" y "Contraseña"**  (Ver nº1 de la captura).
  * **Nota importante:**  Las contraseñas cambiadas en TEMIS **no se sincronizan en Helix de forma inmediata**.
      * La sincronización con Keycloak se realiza cada día a las **22:30 h**.
      * Si un usuario cambia su contraseña, esta **no será válida en Helix hasta el día siguiente**.
      * **OJO!**  En el entorno de "qa" no es posible acceder por usuario TEMIS porque puede que las contraseñas de PRE de TEMIS estén caducadas.
**2.Acceso mediante Certificado Digital**

  * Requiere tener un certificado digital válido instalado en el navegador.
  * El acceso se realiza seleccionando **"proxy-clave"**  (Ver nº2 de la captura).

**3. Acceso mediante LDAP**

  * Usa las credenciales de la VPN.
  * El acceso se realiza seleccionando **"ssoweb"**  (Ver nº3 de la captura).

====== Núcleo Técnico de BMC Helix: Modelo de Datos y Flujo de Información ======
</WRAP>

Este apartado describe los conceptos técnicos fundamentales necesarios para comprender el núcleo de **BMC Helix**, centrándose en el diseño del modelo de datos y en el flujo de la información dentro de la plataforma.

===== 1. Diseño del Modelo de Datos: Foundation Data =====

El diseño de BMC Helix se basa en una estructura denominada **Foundation Data (Datos de Base)**.  
Este conjunto de datos constituye el esqueleto común sobre el que se apoyan todos los módulos de la suite (Incidencias, Cambios, Activos, CMDB, etc.).

[[https://extranet.chap.junta-andalucia.es/dokuwiki/!ssdj/doku.php?id=pag_apli:remedy_itsm:problemas#extraccion_de_informes|Extracción de informes]]
==== 1.1 Estructura relacional y multitenencia ====

* El modelo de datos está diseñado para operar en **modo multi-tenancy (multitenencia)**.
* Permite la segregación de información por **compañías u organizaciones** dentro de un mismo sistema.
* Cada registro queda vinculado a una compañía, garantizando aislamiento lógico de los datos.

==== 1.2 Organización jerárquica de los datos ====

Los datos se estructuran de forma jerárquica:

**Compañía**
  * **Ubicaciones**
    * Sitios
    * Edificios
  * **Personas**
  * **Grupos de Soporte**

Esta jerarquía es transversal y es utilizada por todos los módulos funcionales.

==== 1.3 Relación de tablas (formularios) ====

En la arquitectura de BMC Helix, las tablas se denominan técnicamente **formularios**.  
Las relaciones entre ellos son lógicas y funcionales:

Una **Persona**

  * Está asociada a una **Compañía**
  * Está asociada a un **Sitio**
* Un **Grupo de Soporte**
  * Contiene múltiples Personas
* Los **Elementos de Configuración (CI)** de la **CMDB**
  * Se relacionan entre sí
  * Permiten construir mapas de dependencias (por ejemplo, un servidor que soporta una base de datos o una aplicación)

===== 2. Flujo de Información al Crear una Incidencia o Petición =====

Cuando un usuario registra una incidencia o petición, la información sigue un flujo predefinido hacia formularios y repositorios específicos.

==== 2.1 Ubicación física de los datos ====

**Entorno On-Premise**
  * Los datos residen en el **CPD** o infraestructura de la organización

==== 2.2 Formularios implicados en el registro ====

Durante la creación de un ticket intervienen varios formularios clave:

**Datos de la persona**
  * Se consultan y vinculan desde el formulario principal:
    * `CTM:People`
**Datos del ticket**
  * Se almacenan en los formularios del módulo de **Incident Management**
  * El formulario principal es:
    * `HPD:Help Desk`
**Categorización**
  * Uso de:
    * Categorías Operacionales
    * Categorías de Producto
  * Definen el tipo de servicio, aplicación o hardware afectado

==== 2.3 Relaciones automáticas con la CMDB ====

Al crear el ticket:

  -   El sistema intenta asociarlo automáticamente a un **CI**
  -   Esto ocurre si el usuario tiene activos asignados

Esta relación permite:

  -   Analizar impacto
  -   Visualizar dependencias de servicio
  -   Mejorar la gestión de incidencias y problemas

===== 3. Relación de Datos y Visibilidad =====

El acceso a la información está controlado mediante **Restricciones de Acceso (Access Restrictions)**.

==== 3.1 Restricciones por compañía ====

Un usuario solo puede:

  -   Ver
  -   Crear
  -   Modificar información asociada a las compañías autorizadas en su perfil de persona.

==== 3.2 Grupos de permisos por módulo ====

Cada módulo de BMC Helix dispone de **Grupos de Permisos**, entre los más habituales:

  - Master
  - User
  - Submitter
  - Viewer

Estos permisos determinan si el usuario puede:

  - Crear registros
  - Modificar información
  - Consultar datos en modo lectura

===== 4. Información de Referencia =====

Para profundizar en el esquema técnico, relaciones de formularios y arquitectura interna, se debe consultar la documentación oficial de **BMC Helix Subscriber Information**, especialmente el apartado de **Arquitectura**, donde se describen los procesos de:

* Activation
* Onboarding
* Ciclo de vida de los servicios

> Nota: El enlace exacto puede variar según la versión de Helix, pero este portal constituye la referencia oficial para la documentación técnica y funcional.

====== Modelo de Datos BMC Helix ======

El siguiente esquema representa el **modelo lógico de datos de BMC Helix**, mostrando la relación entre la compañía (tenant), los datos maestros (Foundation Data) y los módulos operativos.

<code>
                         COMPAÑÍA
                    Dato raíz / Tenant
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
   UBICACIONES           PERSONAS              GRUPOS DE SOPORTE
   Sitios / Edificios    CTM:People            Equipos resolutores
        │                     │                     │
        └───────────────┬─────┴───────────────┬─────┘
                        │                     │
                        ▼                     ▼
              INCIDENCIA / PETICIÓN      CATEGORIZACIÓN
                 HPD:Help Desk           Operacional / Producto
                        │
        ┌───────────────┼────────────────┐
        │               │                │
        ▼               ▼                ▼
      CMDB             SLM          HISTÓRICO / NOTAS
      CI afectado      SLA / ANS    Seguimiento del ticket
</code>

===== Notas =====

  - La **Compañía** actúa como elemento raíz y define el ámbito de visibilidad de los datos.
  - **Ubicaciones**, **Personas** y **Grupos de Soporte** forman parte del **Foundation Data**.
  - Las **Incidencias y Peticiones** se gestionan desde el formulario `HPD:Help Desk`.
  - La **CMDB** permite relacionar el ticket con el **CI afectado**.
  - El módulo **SLM** controla los compromisos de **SLA / ANS**.
  - El **histórico y las notas** garantizan la trazabilidad completa del ticket.

</panel>
