
# Reinicio Cluster Kubernetes

## Reinicio - Maquina de Salto

Alguna ves tendremos que reiniciar los nodos de un cluster de Kubernetes para que cojan cierta configuración, lo pasos serían los siguientes:

***Este proceso debe realizarse nodo por nodo, nunca todos a la vez***

***Si alguna ves al cambiar el fichero config de .kube no nos reconoce el cluster debemos de volver a descargar el fichero kubeconfig del rancher y pegarlo de nuevo en nuestro respectivo config, para después copiarlo en el fichero config***

1. Antes de reiniciar un nodo, se debe parar la entrada de procesos a dicho nodo:
`kubectl cordon <nombre-nodo>`
2. Después movemos los pods a otros nodos. Esto lo haces drenando el nodo, lo que migra los pods sin interrumpir los servicios, se lanza el siguiente comando desde la máquina de salto:
`kubectl drain <nombre-nodo> --ignore-daemonsets --delete-emptydir-data`
3. Una vez echo lo anterior te metes en el nodo en concreto que hiciste el drenado y lo reinicias.
`reboot`
4. Una vez reiniciado, de nuevo en la máquina de salto, ejecutamos el siguiente comando para volver a habilitarlo para que Kubernetes lo use nuevamente.
`kubectl uncordon <nombre-nodo>`

## Reinicio - Desde un mismo Nodo

En algunos casos es aconsejable realizar los comandos de kubernetes desde dentro del mismo nodo.

```jsx
/var/lib/rancher/rke2/bin/kubectl --kubeconfig /var/lib/rancher/rke2/server/cred/admin.kubeconfig get nodes
```

Para ejecutar cualquier comando de Kubernetes en el nodo, pondríamos la ruta de arriba seguida del comando, por ejemplo: “get nodes”, “cordon”, “uncordon”.

# Creación Rancher por Comando

## Preparación Inicial de la máquina

Actualizamos la máquina e instalamos algunos paquetes:

`zypper refresh   zypper update -y  reboot  zypper search-packages git  SUSEConnect -p sle-module-desktop-applications/15.5/x86_64  SUSEConnect --product sle-module-development-tools/15.5/x86_64  zypper in git`

## Instalar Kubernetes con K3s

Seguimos los siguientes pasos:

`curl -L https://github.com/k3s-io/k3s/releases/latest/download/k3s -o k3s  chmod +x k3s  mv k3s /usr/local/bin/  mkdir -p /etc/rancher/k3s  vi /etc/systemd/system/k3s.service`

Introducimos el siguientes contenido en el documento:

```bash
[Unit]
Description=Lightweight Kubernetes
Documentation=https://k3s.io
After=network-online.target
Wants=network-online.target

[Service]
Type=notify
ExecStart=/usr/local/bin/k3s server
KillMode=process
Delegate=yes
LimitNOFILE=1048576
LimitNPROC=infinity
LimitCORE=infinity
TasksMax=infinity
TimeoutStartSec=0
Restart=always

[Install]
WantedBy=multi-user.target
```

## Instalar helm

Instalamos helm con el siguiente procedimiento:

Deshabilita Proxy y recargamos
`curl -LO https://get.helm.sh/helm-v3.15.4-linux-amd64.tar.gz #La ultima actual estable en nuestro caso, en el futuro revisar#`

`tar -zxvf helm-v3.15.4-linux-amd64.tar.gz`

`mv linux-amd64/helm /usr/local/bin/helm`

`helm version - Para confirmar`

`helm repo add rancher-latest https://releases.rancher.com/server-charts/latest`

`helm repo add jetstack https://charts.jetstack.io`

`helm repo list`

`helm repo update`

`k3s kubectl create namespace cattle-system`

`export KUBECONFIG=/etc/rancher/k3s/k3s.yaml`

`helm upgrade --install cert-manager jetstack/cert-manager --namespace cert-manager --create-namespace --version v1.18.2`

`k3s kubectl get pods -n cert-manager - Para confirmar funcionamiento`

`hostname="hostname".justicia.junta-andalucia.es --set bootstrapPassword=”Contraseña Web rancher”`

## Error al crear Clústeres nuevas / Añadir nuevos nodos en el rancher antiguo

Cuando se ejecuta Rancher con **Podman (o Docker)** sin configurar una red fija, cada vez que se reinicia el contenedor:

- Podman asigna una **nueva IP interna** dentro de su red por defecto (`10.88.0.x`).
- Rancher **usa esa IP interna** para comunicarse con:
- Los nodos (`system-agent`)
- Los agentes de clúster
- El CAPI, RKE2/K3s controllers
- Webhooks internos

Cuando esa IP cambia:

❌ Los agentes de los nodos siguen intentando conectar a la **IP anterior**

❌ El servidor Rancher pierde visibilidad interna con el clúster de gestión (K3s)

❌ El plan de provisión no puede conectar con los nodos

❌ Se generan continuamente errores como:

```
invalid bearer token, Token has been invalidatedno route to host
failed to create cluster accessor
dial tcp 10.43.xx.xx:443: no route to host
```

Por eso **los nodos nuevos nunca pasaban a Ready o “no se podían crear nuevos clústeres”**, y algunos existentes tampoco.

Para solucionar el problema, desde la máquina del rancher hay que tirar los siguientes comandos:

```bash
#Sirve para ver la IP del contenedor.
podman inspect -f '{{.NetworkSettings.IPAddress}}' rancher

#Sirve para entrar en el contenedor.
podman exec -it rancher bash

#Sirve para ver la IP del endpoint.
kubectl get endpoints -n cattle-system rancher -o yaml

#Sirve para editar el fichero del endpoint.
kubectl edit endpoints rancher -n cattle-system
```

Para que funcionara tendríamos que poner la IP 10.88.0.24 dentro del fichero sustituyendo la IP 10.88.0.3.

Tras esto ya se pueden crear clústeres y añadir nodos.

- ****OJO, si reiniciamos de nuevo el contendor de podman, la ip del contenedor volverá a cambiar y habrá que realizar estos pasos con la nueva IP*****

## Script para descargar los Kubeconfig de todos los ranchers

“actualizar_todos_ranchers.sh”

```bash
#!/bin/bash

# ==============================================================================
# VARIABLES DE COLOR
# ==============================================================================
greenColour="\e[0;32m\033[1m"
endColour="\033[0m\e[0m"
redColour="\e[0;31m\033[1m"
blueColour="\e[0;34m\033[1m"
yellowColour="\e[0;33m\033[1m"
purpleColour="\e[0;35m\033[1m"
turquoiseColour="\e[0;36m\033[1m"
grayColour="\e[0;37m\033[1m"

# ==============================================================================
# CONFIGURACIÓN INICIAL
# ==============================================================================
TARGET_DIR="/root/.kube"
FECHA=$(date "+%Y-%m-%d %H:%M:%S")

# Usamos 'echo -e' para que interprete los colores
echo -e "${purpleColour}>>> Iniciando actualización de clústeres:$FECHA${endColour}"

# ------------------------------------------------------------------------------
# FUNCIÓN MAESTRA (NO TOCAR)
# Descarga el kubeconfig de cualquier clúster usando la API
# ------------------------------------------------------------------------------
actualizar_cluster() {
    local RANCHER_URL=$1
    local TOKEN=$2
    local CLUSTER_ID=$3
    local NOMBRE_FICHERO=$4

    # Añadimos "Bearer" si falta
    if [[ "$TOKEN" != "Bearer "* ]]; then TOKEN="Bearer$TOKEN"; fi

    # Imprimimos el inicio de la línea (sin salto de línea con -n)
    echo -ne "${grayColour}   -> Bajando${blueColour}$NOMBRE_FICHERO${endColour}${grayColour} ($CLUSTER_ID)...${endColour}"

    # Petición a la API (Silenciosa)
    RESPONSE=$(curl -s -k -X POST \
        -H "Authorization:$TOKEN" \
        -H "Content-Type: application/json" \
        "$RANCHER_URL/v3/clusters/$CLUSTER_ID?action=generateKubeconfig")

    # Validación
    if echo "$RESPONSE" | grep -q "baseType"; then
        echo "$RESPONSE" | jq -r '.config' > "$TARGET_DIR/$NOMBRE_FICHERO"
        chmod 600 "$TARGET_DIR/$NOMBRE_FICHERO"
        # OK en VERDE
        echo -e "${greenColour}OK${endColour}"
    else
        # ERROR en ROJO
        echo -e "${redColour}ERROR${endColour}"
        # Detalle en AMARILLO

}

# ==============================================================================
# RANCHER PRE-PRODUCCIÓN - 1 (iusrancher-pre)
# ==============================================================================

URL_2="https://iusrancher-pre.justicia.junta-andalucia.es"
TOKEN_2="token-dnb28:xx8wqcg2bgnbld4gdxnkrz8zrljm8m66fvhtczwf9v4njwk2kbg54h"

# ==============================================================================
# RANCHER PRODUCCIÓN - 1 (iusrancher)
# ==============================================================================
```
