**OBSERVACIONES

  * El Schedule //BBDD_SEDES//  es un Incremental todos los días.
  * La configuración de las retenciones y Schedule recaen sobre el grupo y no sobre el cliente
  * Todos los grupos, salvo IUSNAS01 que está en desuso, están asociados al Media Pool //PoolDD2500//, lo que implica que se salva todo al Data Domain 2500 IUSAPBACK01. Dicho Data Domain tiene una tarea programada de limpieza de datos expirados que se lanza los Lunes a las 11:15
  * La clonación de reciclado de cintas se realizan bajo tareas cron por scripts desde el servidor de Networker Oragrid.
  * De la misma forma, por script mediante tareas cron, se realizan monitorizaciones de espacio en cintas, tamaño de copias, clientes, cruce de Save sets para TPath2,...
  * Las cintas se guardan en el armario ignifugo del Pre-CPD
  * Las políticas de reintentos de backup después del fin de semana son para los Comunes (full) y Adriano (full y level 5).
  * Este servidor además de las copias de las Sedes también tiene configurado el cliente iusnas05 para realizar copias de la ISILON, concretamente los save sets /ifs/data/SMB-zone/RecibidosRepprinst/, RecibidosRepjinst, RecibidosRestoRepartos, RecibidosJinst, RecibidosPrinst y RecibidosResto, de las distintas provincias.
  * Existe un documento donde se guardan la configuración completa, clientes y un bitácora de los cambios de esta en:

//\\iusnas05.justicia.junta-andalucia.es\SIJ\Sistemas\GTAB\Inventario de Oragrid.ods.//
