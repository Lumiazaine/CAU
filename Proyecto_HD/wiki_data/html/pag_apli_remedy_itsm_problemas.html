Procedimientos y manuales para técnicos de BMC HELIX (REMEDY).

Configuración de usuario, licencias de usuario y permisos BMC Helix

Esta información describe los distintos tipos de licencias en BMC Helix ITSM y su aplicación en la configuración de accesos.

Incluye:

  • Diferencias entre licencias Suite y Service Desk.
  • Requisitos de permisos por aplicación.
  • Casos prácticos de asignación de permisos desde la pestaña *“Permiso de Aplicación”*.

Nota: Este contenido está orientado a personal técnico y administradores funcionales.

Tipos de Licencias

Suite User License:

Dirigido a: perfiles superiores como administradores de la plataforma y personal que gestiona múltiples módulos (incidentes, cambios, activos, CMDB, etc.).

Ruta: Consola de Administración → Personas → Detalles de inicio de sesión/acceso`

Configuración:

  • Tipo de Licencia: `En paquetes`
  • Tipo de licencias agrupada: `Agregar Agrupada` → seleccionar 2015 Pricing Model
  • Elegir: Remedy Service Management Suite - User License

Esto asigna una licencia a un usuario nombrado (*Named User*) con acceso completo a los módulos principales de la suite ITSM de BMC.

Licencia Flotante:

Remedy Service Management - Concurrent User License` → compartida entre varios usuarios según disponibilidad. ⚠️ No aplica a la suite completa.

Importante: Tener una licencia Suite no implica permisos automáticos. Es necesario configurar los Permisos de Aplicación de forma individual.

Remedy Service Desk:

Dirigido a: Roles de soporte técnico y agentes que gestionan incidencias o solicitudes. Tienen acceso operativo diario, pero no a todos los módulos de la suite completa.

Tipos disponibles:

  • User License (Fija): garantiza que ese técnico tenga siempre una licencia asignada.
  • Concurrent User License: licencias flotantes, compartidas entre varios técnicos.

Al igual que las licencias Suite, requieren asignar los Permisos de Aplicación específicos.

¿Cómo se clasifican y asignan los permisos por módulo en BMC Helix - ITSM?

Este artículo proporciona una guía estructurada de los permisos y roles disponibles en BMC Helix ITSM, organizados por módulo funcional.

Incluye:

  • Clasificación por tipo de acceso (User, Viewer, Config, Admin, etc.)
  • Descripción de los permisos mínimos recomendados para un técnico de soporte
  • Descripción de los módulos con roles especiales.

Orientado a: Administradores del sistema, Gestores de procesos y Equipos de Soporte que necesiten comprender, asignar o revisar permisos de forma controlada y eficiente.

1. Tipos de permisos generales

Esta tabla resume los tipos de permisos más comunes que se repiten en la mayoría de los módulos de BMC Helix ITSM, descritos en el punto 3. Cada módulo puede tener variantes como *User, Viewer, Config, Admin*, etc., que siguen un patrón similar y que se describen a continuación:

(Esta clasificación permite entender rápidamente el nivel de acceso asociado a cada rol sin necesidad de repetir definiciones en todos los módulos.)

Tipo de Permiso Descripción Breve
User Gestiona elementos asignados o creados (tickets, tareas, solicitudes..)
Viewer Acceso en sólo lectura
Submitter Puede crear registros, pero no gestionarlos
Config Permite configurar aspectos funcionales del módulo
Admin / Administrator Acceso completo: configuraciones globales, reglas, flujos
Master / Manager / Coach Gestión total o supervisión de tickets o procesos

2. Permisos mínimos para un técnico de soporte

Estos son los permisos operativos básicos recomendados para perfiles de soporte técnico (como CAU, Nivel 1 o Nivel 2). Permiten trabajar con incidencias, solicitudes, tareas, órdenes de trabajo y consultar activos.

(La asignación de estos roles garantiza que el técnico pueda ejecutar su trabajo diario sin exponer configuraciones críticas del sistema.)

Permiso Función Principal
Incident User Gestiona incidentes asignados
Service Request User Crear y consultar peticiones de servicio
Task User Trabajar tareas asignadas
Work Order User Ejecutar órdenes de trabajo asignadas
Asset Viewer Consultar activos y su información relacionada

3. Permisos por módulo

En esta sección se listan los módulos principales de BMC Helix ITSM, diferenciando en cada uno de ellos por tipo de permisos:

  • Permisos generales mencionados en el nº1 con carácter general.
  • Permisos específicos adicionales por cada módulo.

3.1 ACTIVITY:

Este módulo está relacionado con el registro y seguimiento de actividades dentro de tickets, es decir la gestión asociada a los tickets como tareas dentro de incidencias, cambios, etc.

  • Permisos generales mencionados anteriormente: `Activity User`, `Activity Viewer`, `Activity Config`.

3.2 AR SYSTEM (orientado a administradores técnicos y desarrollo):

Estos permisos están más orientados a usuarios con roles técnicos o de administración avanzada que gestionan formularios, estructuras y funcionamiento interno del sistema.

  • Administrator: Acceso total al sistema (formularios, workflows, reglas).
  • Browser: Solo lectura.
  • Customize: Modifica vistas personales.
  • Struct Admin / Subadmin: Modifica o crea objetos técnicos del sistema.
  • Sub Administrator: Admin limitado a ciertas áreas (sin acceso estructural).

3.3 ASSET:

Se refiere a la gestión de activos dentro de BMC Helix ITSM. Este módulo se usa para el control de inventario, configuración de elementos de infraestructura (CI) y su ciclo de vida.

  • Todos los permisos siguen el esquema general: `Admin`, `Config`, `User`, `Viewer`.

3.4 CHANGE:

Enfocado en la gestión del cambio (*Change Management*) dentro de ITSM.

  • Permisos generales: `Change Config`, `Change Master`, `Change User`, `Change Viewer`.
  • Permiso específico: `Infrastructure Change Submit` → solo permite crear solicitudes de cambio (RFCs).

3.5 FINANCIAL:

Módulo enfocado al análisis de costes asociados a servicios o activos.

  • Cost Manager: Gestiona y analiza los costes de servicios o activos.

3.6 FOUNDATION:

Aquí se definen estructuras organizativas, usuarios, ubicaciones, roles, reglas de negocio, etc. Es el manejo de datos base de todo el ecosistema ITSM.

  • Approval Admin: Administra reglas de aprobación.
  • Config Categorization Admin: Configura estructura Categoría/Tipo/Producto.
  • Contact Admins (People, HR, Location, Org): Gestionan usuarios y datos organizativos.
  • DMT Admin / User: Importación masiva de datos.
  • DSL Master / User / Viewer: Gestión del catálogo de servicios.
  • Cognitive Service Config: Configuración de clasificación automática (IA).
  • Progressive Views Config: Diseño de formularios modernos (Smart IT).
  • Email Rule Config: Reglas de automatización por correo.
  • Security / Notification / Licensing Admin: Gestión de acceso, licencias y notificaciones.

3.7 IDENTITY REQUEST (IRM):

Forma parte de IRM (Identity Request Management) en BMC Helix. Este módulo se enfoca en gestionar solicitudes relacionadas con accesos, identidades, perfiles y autorizaciones.

  • IRM Admin: Control total del sistema de solicitudes.
  • IRM Configuration Manager: Define catálogos, atributos y políticas.
  • IRM Delegated Admin: Admins con alcance restringido por unidad.
  • IRM Manager: Aprueba o rechaza solicitudes.
  • IRM Request Submitter / IRM User: Crean y hacen seguimiento de solicitudes.

3.8 INCIDENT:

Este módulo se usa para el registro, seguimiento, resolución y cierre de incidencias (según ITIL).

  • Permisos generales: Incident Config, Incident Master, Incident User, Incident Viewer.
  • Permisos específicos:

-Compliance Sensitive Data View: Permiso para acceder a datos sensibles. -Incident Submitter: Puede crear tickets, pero no gestionarlos.

3.9 KNOWLEDGE (alineado a KCS):

Gestiona la base de conocimiento (*Knowledge Base*) en BMC Helix ITSM, alineado con prácticas KCS (Knowledge-Centered Service).

  • Permisos generales: Knowledge Admin, Knowledge Config, Knowledge User, Knowledge Viewer.
  • Permisos específicos:

-KCS Candidate: En formación, crea borradores.

-KCS Contributor: Crea y edita artículos (requiere aprobación).

-KCS Publisher: Publica artículos directamente.

-KCS Coach: Supervisa y aprueba contenido.

-Knowledge Config: Configura aspectos funcionales del módulo.

-Knowledge Submitter / User / Viewer: Participa en el ciclo de vida o consulta artículos.

3.10 MyIT/DWP:

En este módulo, ahora conocido como *Digital Workplace*, el permiso visible es:

  • MyIT Admin: Administra el portal de autoservicio (servicios, catálogos, reglas de visibilidad). Es esencial para gestores del portal de usuario o equipo de experiencia digital.

3.11 PROBLEM:

Encargado de la gestión de problemas dentro del ciclo de vida ITIL. Se usa para identificar causas raíz, documentar workarounds y prevenir recurrencia de incidentes.

  • Permisos generales: `Problem Config`, `Problem Master`, `Problem User`, `Problem Viewer`.
  • Permiso específico:

-Problem Submitter: Crea problemas, sin acceso a gestión posterior.

3.12 PROCESS DESIGNER:

Permite modelar flujos de trabajo (workflows) para ITSM: incidentes, cambios, tareas, etc., mediante lógica visual y automatizaciones.

  • Process Designer Admin: Crea y publica workflows.
  • Process Designer Sub Admin: Puede editar procesos asignados (sin publicar).

3.13 RELEASE:

Orientado a la gestión de versiones y despliegues dentro del ciclo de vida ITIL (Release Management). Es clave para planificar, coordinar y desplegar cambios en entornos de producción.

  • Permisos generales: Release Config, Release Master, Release User, Release Viewer.

3.14 REQUEST/SRM:

Gestiona las peticiones de servicio (*SRM - Service Request Management*) en BMC Helix. Incluye Work Orders, flujos guiados de solicitud y autoservicio.

  • Permisos generales: `Work Order User`, `Work Order Viewer`.
  • Permisos específicos:

-Business Analyst: Diseña y prueba flujos de solicitud.

-Business Manager: Supervisa métricas y SLAs.

-Entitlement Administrator: Define visibilidad y acceso.

-Request Catalog Manager: Administra el catálogo de servicios.

-Service Request User: Usuario general de SRM (crear y consultar solicitudes).

-SRM Administrator: Configuración total del módulo SRM.

-Work Order Config: Configura el módulo de Work Orders.

-Work Order Master: Gestión completa de Work Orders.

-Work Order Submitter: Crea solicitudes sin capacidad de edición posterior.

3.15 ROI:

Se refiere al Return on Investment Dashboard o análisis de valor para ITSM. Más orientado a la medición de impacto y eficiencia del uso de la plataforma.

  • ROI Admin: Configura el módulo de métricas.
  • ROI Viewer: Consulta dashboards.

3.16 SLM (Service Level Management):

Permite definir, medir y supervisar acuerdos de nivel de servicio (*SLAs, OLAs y UCs*) en BMC Helix ITSM.

  • Permisos generales: SLM Config.
  • Permisos específicos:

-SLM Console and Dashboard User: Visualiza métricas de SLAs.

-SLM Customer: Consulta SLAs asignados.

-SLM Manager / Unrestricted Manager: Gestiona acuerdos activos (con/sin restricciones).

3.17 TASK:

Permite gestionar tareas (Tasks) asociadas a otros procesos como Incidents, Changes, Work Orders, etc. Este sistema de tareas ayuda a descomponer el trabajo en pasos individuales, asignables y rastreables.

  • Permisos generales: `Task Administrator`, `Task User`, `Task Viewer`.
  • Permisos específicos:

-Task Application Config: Configura tareas según módulo.

-Task Process Config: Define flujos de tareas.

-Task Manager: Puede reasignar y gestionar todas las tareas.

3.18 VIRTUAL AGENT (chatbot y soporte conversacional) — NO APLICA

BLOQUE I – Explotación del Sistema

En este bloque se describen las tareas y procedimientos que deben seguirse para asegurar el funcionamiento óptimo del sistema, incluyendo políticas de backup, restauración, rotación de logs, paradas programadas y operaciones de carga y descarga de datos.

<panel type=“info”>

1.1 Políticas de Backups, Rotación de Logs y Demás Tareas Programadas

En este epígrafe se abordan las recomendaciones de programación de backups, tanto de base de datos como de máquinas virtuales, así como las políticas de rotación de logs y otras tareas periódicas de mantenimiento.

1.1.1 Política de Backups

Sistema: Base de datos del entorno BMC Helix On-Premise Base de Datos: Oracle Database Utilidad: Herramienta de backup de base de datos (por ejemplo, *Oracle Recovery Manager – RMAN*)

Estrategia de Backup: La estrategia se basa en una combinación de copias de seguridad completas (full) e incrementales, con el fin de optimizar el tiempo de ejecución y el espacio de almacenamiento.

1.1.1.1 Política de Backups de Base de Datos

Backup Completo (Full):

  • Frecuencia: Cada fin de semana.
  • Horario: Durante la ventana de mantenimiento programada, cuando la actividad de los usuarios es mínima (por ejemplo, sábado o domingo por la mañana).
  • Método: Se utilizará RMAN para realizar una copia de seguridad completa de los *datafiles*, el *control file* y el *spfile*.
  • Realización: El backup puede realizarse en caliente.
  • Período de Retención: Se conservarán las últimas 4 copias de seguridad completas (equivalentes a 1 mes de histórico).

Backup Incremental Diario:

  • Frecuencia: De lunes a viernes.
  • Horario: Programado fuera del horario laboral pico (por ejemplo, después de la medianoche).
  • Método: RMAN realizará una copia de los bloques de datos modificados desde el último backup de nivel 0.
  • Realización: Puede realizarse en caliente.
  • Período de Retención: 30 días.

1.1.1.2 Política de Backups de Máquinas Virtuales (VMs)

Copia de Seguridad del Plano de Control y Nodos Master

Elemento Frecuencia Retención Mecanismo Justificación
Estado del Clúster (etcd) Diario 15 días `etcdctl snapshot save` Garantiza la consistencia transaccional. Permite reconstruir el clúster en caso de falla.

Este procedimiento se ejecuta únicamente en uno de los nodos de control (Master Node).

Procedimiento:

1. Definir la ruta del binario

bash # Buscar la ruta al binario crictl sudo find / -name crictl # Definir la ruta completa del binario crictl obtenida en el paso anterior CRICTL_PATH="/var/lib/rancher/rke2/data/v1.28.15-rke2r1-cfade09fe202/bin/crictl" # Definir variables TIMESTAMP=$(date +%Y%m%d%H%M) SNAPSHOT_FILE="/var/lib/etcd/etcd-snapshot-$TIMESTAMP.db"

2. Identificar el ID del contenedor etcd

bash ETCD_ID=$(sudo $CRICTL_PATH ps -a | grep etcd | grep Running | awk '{print $1}' | head -n 1) echo "ID del Contenedor de etcd: $ETCD_ID"

3. Tomar Snapshot dentro del contenedor

bash sudo $CRICTL_PATH exec -i $ETCD_ID etcdctl snapshot save $SNAPSHOT_FILE

4. Extraer y proteger el archivo

bash sudo $CRICTL_PATH cp $ETCD_ID:$SNAPSHOT_FILE /tmp/etcd-backup-$TIMESTAMP.db ls -lh /tmp/etcd-backup-$TIMESTAMP.db scp /tmp/etcd-backup-$TIMESTAMP.db usuario_backup@servidor_externo:/backups/etc

Elemento Frecuencia Retención Mecanismo Justificación
Configuración de VM (Nodos Master) Semanal 2 copias Snapshot de la VM completa (hipervisor) Permite restaurar rápidamente el nodo si falla el SO o el disco.

Copia de Seguridad de los Nodos Worker

Elemento Frecuencia Retención Mecanismo Justificación

1.1.2 Procedimiento de Restauración

Este plan describe los pasos a seguir en caso de pérdida total de la base de datos o fallo de las máquinas virtuales.

1.1.2.1 Restauración de Base de Datos

Escenario de Falla: Corrupción de datos o pérdida de datafiles.

Procedimiento de Recuperación:

1. Parar el aplicativo (ver sección 2.2). 2. Abrir el menú de navegación y seleccionar *Oracle Database*. 3. Seleccionar el servicio de base de datos afectado. 4. Escoger el entorno correspondiente:

Entorno Servidor Base de Datos
DEV iusslbddesa05.justicia.junta-andalucia.es desa05
QA iusslbdpre05.justicia.junta-andalucia.es pre05
PROD iusslbdcw01.justicia.junta-andalucia.es procw01

5. Seleccionar “Restaurar” y una de las siguientes opciones:

  1. Restaurar a la última versión.
  2. Restaurar a un registro de hora.
  3. Restaurar a SCN (System Change Number).

6. Ejecutar la restauración y validar el proceso. 7. Realizar pruebas de conexión y verificación de datos. 8. Informar a los equipos de negocio y aplicaciones sobre la disponibilidad del servicio.

1.1.2.2 Restauración de Máquinas Virtuales

Restauración de un Nodo Worker

1. Aislar el nodo fallido: 

 %%bash kubectl cordon <nombre-del-nodo> %%

2. Apagar la VM fallida desde vSphere. 3. Restaurar el snapshot deseado. 4. Iniciar la VM y verificar su estado: bash kubectl get nodes 5. Desbloquear el nodo: bash kubectl uncordon <nombre-del-nodo>

Restauración de un Nodo Master

⚠️ La restauración de un nodo de control es sensible, ya que contiene la base de datos distribuida *etc*.

Escenario A – Recuperación del SO de un solo Maestro:

  1. Apagar la VM afectada.
  2. Restaurar snapshot.
  3. Verificar que el directorio `/var/lib/rancher/rke2/server/db/etcd` no haya sido revertido.
  4. Iniciar la VM; RKE2 se sincronizará automáticamente con los demás nodos.

Escenario B – Falla Catastrófica (Pérdida de Quórum):

  1. Apagar todos los nodos maestros.
  2. Copiar el último backup `.db` de etcd al nodo a restaurar.
  3. Detener el servicio RKE2: bash sudo systemctl stop rke2-server
  4. Ejecutar restauración: bash sudo rke2 server --cluster-reset --cluster-reset-restore-path=/tmp/etcd-backup-YYYYMMDDHHMM.db
  5. Eliminar la bandera *cluster-reset* del archivo `/etc/rancher/rke2/config.yaml`.
  6. Iniciar RKE2: bash sudo systemctl start rke2-server
  7. Comprobar el clúster y reactivar los nodos restantes: bash kubectl get nodes

1.1.3 Rotación de Logs. Periodicidad

En BMC Helix ITSM está configurada la rotación automática de logs, la cual puede revisarse y modificarse desde el formulario:

Ruta:

Aplicaciones > AR System Administration > AR System Administration Console > Server > General > Server Information > pestaña Log Files

En la parte inferior del formulario se configuran los siguientes parámetros:

  • Tamaño máximo del archivo de log.
  • Número máximo de backups a mantener.

1.2.1 Arranque, Rearranque y Parada del Sistema

Parada del sistema

El entorno de BMC Helix está diseñado para operar 24×7. El despliegue sobre Kubernetes no está pensado para “apagarse” completamente, ya que la destrucción de un *pod* genera automáticamente otro *pod* idéntico para mantener el servicio.

Si se desea parar el entorno completamente, es necesario poner a cero la escala de los Deployments y StatefulSets de los namespaces `helix-is` y `helix-common`.

Conceptos clave:

  • Deployment: recurso de Kubernetes para gestionar aplicaciones *stateless* (sin estado).
    1. Mantiene el número deseado de réplicas.
    2. Reemplaza pods de forma controlada (rolling update / rollback).
    3. Todos los pods son idénticos y reemplazables.
  • StatefulSet: recurso para aplicaciones *stateful* (con estado).
    1. Asigna nombre único y estable a cada pod (por ejemplo `redis-0`, `redis-1`).
    2. Mantiene orden de creación y terminación.
    3. Soporta volúmenes persistentes independientes (PVCs).

Para detener completamente el despliegue:

  • Poner a cero todos los valores de réplicas (`replicas: 0`) de los Deployments y StatefulSets.
  • Con ello los pods se destruirán y no se volverán a crear automáticamente.

Inicio ordenado

Si el entorno ha sido detenido por completo, el orden correcto para el arranque es el siguiente:

1. Devolver el número de réplicas original a todos los Deployments y StatefulSets del namespace `helix-common`. 2. Esperar a que todos los pods estén en estado `Running`. 3. En el namespace `helix-is`, devolver las réplicas a los StatefulSets `platform-*`. 4. Esperar entre 7 y 8 minutos hasta que estén completamente operativos. 5. Devolver el número de réplicas original al resto de Deployments y StatefulSets del namespace `helix-is`.

Reinicio ordenado

El reinicio implica la destrucción (borrado) y recreación automática de los pods. Debe realizarse en un orden concreto para evitar inconsistencias en los servicios dependientes.

1.3 Carga y Descarga de Datos Masiva

1.3.1 Crear archivos .arx, .xml y .csv con artext

La utilidad artext permite generar archivos en formato .arx, .xml o .csv para ser utilizados en operaciones de importación y localización de datos dentro de AR System.

Formatos soportados:

  • AR Export (.arx) – formato nativo de AR System.
  • AR XML (.xml) – archivo XML que cumple con la especificación AR XML.
  • CSV (.csv) – archivo de valores separados por comas.
  • ASCII (.asc) – archivo de texto plano genérico.

Recomendaciones al crear archivos ASCII externos:

  • Usa una cadena separadora única (hasta 32 caracteres).
  • Evita que el separador aparezca en nombres de campo o datos.
  • Los caracteres especiales no se aceptan como separadores de columnas.

Archivos adjuntos exportados en formato AR Export: Al exportar desde un navegador, se genera un archivo `.zip` que contiene:

  1. El archivo `.arx`.
  2. Todos los archivos adjuntos exportados.

1.3.2 Exportar un Informe Personalizado de AR System

Puedes guardar o exportar datos desde AR System a distintos formatos para su reutilización en formularios, hojas de cálculo u otras aplicaciones.

Formatos de exportación disponibles:

  • `.arx` – AR Export
  • `.xml` – AR System XML
  • `.csv` – Valores separados por comas

Pasos: 

1. Selecciona el informe en la lista.  
2. En *Destination*, elige si se envía a pantalla, archivo o impresora.  
3. En *Format*, selecciona el formato de salida deseado.  
4. Ejecuta el informe.

Notas:

  • Los adjuntos no se incluyen en archivos CSV (solo se exporta el nombre del archivo).
  • Si eliges destino “Screen” con formato CSV, el informe no se muestra, pero podrás descargarlo.

1.3.3 Importación de Datos en Formularios mediante Data Import

La utilidad Data Import carga datos desde un archivo fuente hacia un formulario de AR System.

Pasos generales: 

1. Ejecutar *Data Import*.  
2. Exportar los datos fuente a un archivo compatible (`.arx`, `.csv`, `.xml`, `.asc`).  
3. Crear un archivo de mapeo (*mapping file*).  
4. Asociar los campos origen y destino.  
5. Guardar y ejecutar la importación.

Campos del cuadro de diálogo “Nuevo Mapeo”:

Campo Descripción
*Source Data File* Archivo de datos a importar (.arx, .csv, .xml, .asc).
*Contains Field Titles* Indica si la primera línea contiene los títulos de los campos.
*Field Separator* Separador de campos (solo formato ASCII).
*Source Form Name* Formulario de origen de los datos.
*Target Server* Servidor de destino.
*Target Form Name* Formulario al que se importarán los datos.
*Custom Options File* Archivo `custom_options.xml` (opcional).

Opciones adicionales:

  • Asociación automática de campos (`Auto Map`).
  • Definición de valores por defecto (*fallback mappings*).
  • Configuración avanzada en pestaña *Options* (tratamiento de fechas, separadores, etc.).

1.3.4 Importación de Datos con DMT (Data Management Tool)

La herramienta Data Management permite importar grandes volúmenes de datos desde versiones anteriores o sistemas externos a BMC Helix ITSM. Se gestiona a través de la consola web *Data Management > Job Console*.

Niveles de datos soportados:

  • *Foundation*
  • *Configuración de procesos (Process Setup)*
  • *Transaccional (Transactional)*

Ventajas principales:

  • Plantillas de trabajo (*Job Templates*) preconfiguradas.
  • Hojas de cálculo y mapeos listos para usar.
  • Ejecución simultánea de múltiples trabajos.
  • Multi-tenencia por compañía.
  • Validación y manejo centralizado de errores.
  • Carga de datos desde varias fuentes (LDAP, AR Export, XML, JDBC, Excel).

Creación de un Job: 

1. En *Data Management > Job Console*, selecciona **Create > New**.  
2. Completa los campos *Job Name*, *Description*, *Company* y *Job Owner*.  
3. Define si se validarán y reemplazarán alias (*Replace Case/Alias = Yes/No*).  
4. Guarda con estado *Draft*.  
5. Una vez creado, puedes programar su ejecución (Schedule Once, Daily, etc.).

Ejecución y Control de Jobs:

  • Desde *Job Console*, selecciona el trabajo y pulsa Run.
  • Los pasos cambian de estado (*Queued → Running → Completed*).
  • Se envía notificación automática al propietario cuando:
    1. El job inicia.
    2. Un paso entra en *Wait* o presenta error.
    3. El job finaliza.
  • Puedes volver a ejecutar (*Re-Run from Validate*) o duplicar el job (*Create as New Job*).

Importante:

  • *Asset Management* no admite importación de SVT o SLA a través de DM.
  • *Service Request Management* usa un método distinto (ver documentación oficial de BMC).

5.3 Instrucciones para la configurar en log del Email Engine de Remedy ITSM

5.3.1 Introducción a BMC ITSM Email Engine

El Remedy Email Engine es un componente de la plataforma BMC Remedy AR System, que se utiliza principalmente para gestionar la comunicación por correo electrónico entre los usuarios y las aplicaciones basadas en Remedy. 

Es un servicio que permite enviar y recibir correos electrónicos desde y hacia el sistema Remedy, de modo que las acciones en la aplicación puedan dispararse o gestionarse vía email.

5.3.2 Funciones principales del Remedy Email Engine

  1. Recepción de correos: Puede procesar emails entrantes y convertirlos en registros dentro de Remedy (por ejemplo, la creación automática de tickets a partir de un correo de soporte).
  1. Envío de notificaciones: Envía correos electrónicos automáticos a usuarios o grupos (ejemplo: notificación de ticket creado, asignación de caso, recordatorios).
  2. Integración flexible: Se conecta con servidores de correo compatibles con protocolos estándar como SMTP, POP3 o IMAP.

5.3.3 Logs del Email Engine

Los logs del Email Engine se almacenan en la siguiente ruta de todos los PODs cuyo nombre es platform-fts-*

Ruta: /opt/bmc/ARSystem/AREmail/Logs

PODs: platform-fts-* 

El fichero **email.log** es donde se almacena el log, pero este fichero está configurado para rotar cuando llega a un tamaño determinado por lo que puede haber múltiples ficheros de log

5.3.4 Cambiar el nivel de log del Email Engine

El nivel de log del email engine se puede modificar para diagnóstico y resolución de problemas:

  • Para cambiarlo, es necesario abrir la consola centralizada desde: AR System Administration > AR System Administration Console > System > General > Centralized Configuration

  • Seleccionar el componente com.bmc.arsys.emaildaemon > platform-fts-xxx

Nota. Las siguientes configuraciones se deben hacer en todos los servidores platform-fts-xxx

  • Identificar el parámetro com.bmc.arsys.emaildaemon.level

  • Cambiar el parámetro con el valor adecuado y pulsar Apply
    • Posibles valores:
  1. OFF
  2. SEVERE (defecto)
  3. WARNING
  4. INFO
  5. CONFIG
  6. FINE
  7. FINER
  8. FINEST
