Categoría de Servicio: Solicitud de acceso

1. Introducción y Alcance

Esta categoría engloba todas las solicitudes de acceso (Peticiones) relacionadas con Altas de Aplicaciones Judiciales, Gestión Procesal y Configuración de Puesto. Su correcta clasificación permite la asignación automática al grupo resolutor adecuado y garantiza la trazabilidad.

Flujo de tareas

Este documento describe el flujo completo de creación automática de tareas (TAE) y la generación de la WO de Soporte Formativo, incluyendo las variaciones para: - Altas de nueva incorporación - Traslados - Perfiles con entrega de equipamiento (portátil/ISL) - Tickets creados por Nivel 1

📘 Diagrama de Flujo 

                         ┌──────────────────────────────────────────┐
                         │              SOLICITUD DE ACCESO         │
                         └──────────────────────────────────────────┘

• El orden de ejecución de las tareas se controla mediante el número de SECUENCIA.

• Las tareas pueden seleccionarse individualmente en función de la necesidad del caso concreto.

• El cierre del ticket puede ser:

  1. Por el usuario (vía Digital Workplace)
  2. Automático a los 15 días
  3. Manual por técnico CAU

📘 Secuencia estándar del proceso

Generación inicial de tareas

1. Creación del ticket

El ticket de Gestión de Usuario puede crearse de dos formas:

  • Desde plantilla de Orden de Trabajo.
  • Sin plantilla (creación manual del ticket).

El comportamiento del sistema y la generación de tareas dependerán de esta elección.

2. Caso 1: Ticket creado desde plantilla

2.1 Generación inicial de tareas

Cuando el ticket se crea desde plantilla, el sistema genera automáticamente las siguientes tareas:

  1. Carga de perfil
  2. Alta de correo del usuario
  3. Solicitud de soporte formativo

El ticket queda asignado inicialmente a CAU.

2.2 Gestión por CAU y cambio de estado

Una vez que CAU realiza la gestión del alta:

  1. Se asigna el ticket se asigna a un usuario técnico de CAU.
  2. El técnico cambia el estado del ticket a “En curso”.

Al pasar a este estado, se activan las tareas.

2.3 Asignación de tareas

Las tareas se asignan de la siguiente forma:

  1. Carga de perfil: Asignada a: DT de la provincia u órgano del usuario
  2. Alta de correo del usuario: Asignada a: CAU
  3. Solicitud de soporte formativo: Asignada a: Capacitación

2.4 Dependencia entre tareas

Estas tareas se activan cuando el técnico de DT cierra la tarea de “Carga de perfil”. No se activan inmediatamente.

  • Alta de correo del usuario
  • Solicitud de soporte formativo

2.5 Paso a estado “Finalizado”

Cuando todas las tareas activas han sido cerradas correctamente, el ticket pasa automáticamente al estado “Finalizado”.

2.6 Perfiles tipo 1 (caso especial)

En el caso de perfiles tipo 1, además de las tareas anteriores, el sistema genera las siguientes tareas adicionales:

  1. Entrega de portátil
  2. Alta ISL

Características de estas tareas:

  • Se activan al mismo tiempo que las tareas de correo y soporte formativo.
  • Se crean en estado “Pendiente”.
  • Se asignan al equipo responsable de la gestión.

2.7 Paso a estado “Finalizado”

Cuando todas las tareas (incluidas las adicionales en perfiles tipo 1) están cerradas, el ticket pasa a estado “Finalizado”.

2.8 Cierre del ticket

Una vez finalizado el ticket: El usuario ya dispone de acceso al portal DIGITAL WORKPLACE.

  1. El usuario puede acceder y cerrar la petición.
  2. Si el usuario no realiza el cierre:
  3. El ticket se cerrará automáticamente a los 15 días.

El técnico de CAU puede cerrar el ticket manualmente una vez finalizada la gestión, en este caso, el usuario no tiene que intervenir en el cierre. Ya se ha validado presencialmente, con la carga de perfil y con la formación, ya sea presencial o remota.

3. Caso 2: Ticket creado sin plantilla

3.1 Comportamiento inicial

Si el ticket se crea sin plantilla: (no recomendable)

No se generan automáticamente las tareas de:

  • Carga de perfil
  • Alta de correo del usuario
  • Solicitud de soporte formativo

El ticket se asigna igualmente a CAU.

3.2 Generación manual de tareas

En este caso CAU debe generar manualmente las tareas desde plantilla.

Una vez creadas las tareas, el procedimiento continúa exactamente igual que en el Caso 1 (ticket con plantilla).

4. Consideraciones operativas

  1. 📌 El orden de ejecución de las tareas puede modificarse cambiando el número de secuencia.
  2. 📌 Es posible seleccionar únicamente las tareas necesarias, en función del caso concreto.
  3. 📌 El cierre del ticket puede producirse por:
  4. 📌 El usuario (vía Digital Workplace)
  5. 📌 Cierre automático a los 15 días
  6. 📌 Cierre manual por técnico de CAU

📘 Flujo de altas de personal externo

Las tareas solo se registran si la petición la generas desde la plantilla de orden de trabajo.

  • Se asigna automáticamente al grupo y usuario técnico responsable.
  • Puedes cambiar el orden de ejecución de las tareas.

Si se crea a mano:

  • Hay que asignar al grupo y responsable ya que por defecto se asigna a CAU.
  • Hay que generar las tareas manualmente.
