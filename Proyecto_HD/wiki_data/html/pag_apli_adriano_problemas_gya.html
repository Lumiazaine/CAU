ERROR USUARIO NO TIENE PERMISO PARA ACCEDER A LA APLICACIÓN

Si en una incidencia indican que el usuario intenta acceder a @driano y le aparece un mensaje en amarillo indicando que el usuario no tiene permiso para acceder a la aplicación:



Hay que comprobar el puesto del usuario, si tiene roles y si el órgano que está dado de alta tiene el rol_cargo correcto, Lo más probable es que en el puesto principal de temis aparece con un cargo que no pertenece a @driano. Un ejemplo sería que aparezca como psicólogo:



Para solucionar el problema hay que ir a mantemis al apartado de Aplicaciones y buscar Gestor procesal adriano:



Marcas el cuadro blanco y pulsas en el botón de la parte inferior que se llama “Perfiles”, pulsamos el cuadro blanco de “Agenda todo permiso (AGE-TODO)” y pulsamos de nuevo en el botón de “Asignar”.

Estos puestos NO SE ASIGNAN en TEMIS, únicamente mediante scripts en CORE_OWNER:

Primero podemos comprobar si el usuario ya dispone de dichos puestos:

1.- Realizamos una consulta del usuario para obtener su ID.

SELECT * FROM CORE_OWNER.USUARIOS WHERE USUARIO='XXXXXXXX';

2.- Añadimos el ID obtenido y realizaremos una consulta que nos muestre todos los cargos de los que disponga que empiecen por "45" ya que los que necesitamos son el "456" y "457".

SELECT * FROM CORE_OWNER.USUARIOS_ORGANOS WHERE ID_USUARIO='XXXXXXXX' AND ID_CARGO LIKE '45%';

3.- Si queremos asegurarnos de que dichos puestos están dados de alta en el Organismo correspondiente, realizaremos la siguiente consulta con el ID_ORGANO obtenido en el segundo paso.

SELECT * FROM CORE_OWNER.ORGANOS WHERE ID = 'XXXXXXXX';

MERGE INTO CORE_OWNER.TM_CODIGOS_POSTALES USING DUAL ON (ID_MUNICIPIO=(SELECT ID FROM CORE_OWNER.TM_POBLACIONES WHERE POBLACIONES='CODIGO_POBLACION') AND CODIGO_POSTAL='CODIGO_POSTAL' ) WHEN NOT MATCHED THEN INSERT (ID,CODIGO_POSTAL,ID_MUNICIPIO,FECHA_INICIO) VALUES (CORE_OWNER.SEQ_TM_CODIGOS_POSTALES_ID.NEXTVAL,'CODIGO_POSTAL', (SELECT ID FROM CORE_OWNER.TM_POBLACIONES WHERE POBLACIONES='CODIGO_POBLACION'),SYSDATE);

SINCRONIZACIÓN DE USUARIOS Y PUESTOS

Una vez que se han realizado modificaciones de usuarios y sus puestos, es necesario lanzar la sincronización con @driano desde la base de datos de CORE_OWNER:

alter session set current_schema=core_owner;
begin
– llamar a todos los métodos de PKG_SINC_TEMIS.USUARIOS, limpieza, sincronización, volcado
PKG_SINC_TEMIS.USUARIOS_INICIO();
PKG_SINC_TEMIS.USUARIOS_SINCRONIZACION_TEMIS_TOTAL();
PKG_SINC_TEMIS.USUARIOS_VOLCADO_ADRIANO();
end;
/
– USUARIOS_ORGANOS
begin
– llamar a todos los métodos de PKG_SINC_TEMIS.USUARIOS_ORGANOS, limpieza, sincronización, volcado
PKG_SINC_TEMIS.USUARIOS_ORGANOS_INICIO();
PKG_SINC_TEMIS.USUARIOS_ORGANOS_SINCRONIZACION_TEMIS_TOTAL();
PKG_SINC_TEMIS.USUARIOS_ORGANOS_VOLCADO_ADRIANO();
end;
/

Con esta sincronización, los datos de los usuarios y sus puestos pasarán a las tablas de CORE_OWNER.

ERRORES Y SOLUCIONES EN LA SINCRONIZACIÓN

Hay ocasiones en las que al hacer la sincro de USUARIOS_ORGANOS aparece el siguiente mensaje:

Informe de error -
ORA-00001: restricción única (CORE_OWNER.UK_USUARIOS_ORGANOS) violada
ORA-06512: en “CORE_OWNER.PKG_SINC_TEMIS”, línea 1916
ORA-06512: en línea 9
00001. 00000 - “unique constraint (%s.%s) violated”
*Cause:An UPDATE or INSERT statement attempted to insert a duplicate key.
For Trusted Oracle configured in DBMS MAC mode, you may see
this message if a duplicate entry exists at a different level.
*Action: Either remove the unique restriction or do not insert the key.

Para solventar este error hay que buscar en primero en CORE_OWNER los registros que dan el problema en la tabla SINC_USUARIOS_ORGANOS:

select * from core_owner.SINC_USUARIOS_ORGANOS stmp1
where exists(select 'T' from core_owner.SINC_USUARIOS_ORGANOS stmp2
where stmp1.ID_USUARIO=stmp2.ID_USUARIO
and stmp1.ID_ORGANO=stmp2.ID_ORGANO
and stmp1.ID_CARGO=stmp2.ID_CARGO
and stmp1.FECHA_INICIO=stmp2.FECHA_INICIO
and stmp1.FECHA_FIN=stmp2.FECHA_FIN
and stmp1.ID!=stmp2.ID); \\
Con el resultado de registros, tomamos los ID y eliminamos los registros en CORE_OWNER.USUARIOS_ORGANOS con ese ID:

delete from core_owner.usuarios_organos where id in ( ‘ID1’ ,’ID2’,..);

Procedimiento para incidencias con procuradores

1. Roles implicados

CAU

  1. Recibe la incidencia o el correo.
  2. Identifica si existe duplicidad o error visible.
  3. Asigna si procede.

Basal (Explotación Adriano)

  1. Responsable final de revisar duplicidades, errores de sincronización, códigos externos y actualizaciones de datos.
  2. Procesal/ Oficina Técnica de Adriano
  3. Si el problema afecta a distribución o esquemas, puede intervenir.

2. Procedimiento operativo para CAU

2.1. Recepción de la incidencia

  1. Registrar el ticket procedente de cualquier canal.
  2. Indicar al emisor que las comunicaciones deben realizarse al correo oficial de CSU (si procede).

2.2. Verificación inicial

  1. Consultar si el procurador aparece en Adriano: CAU no tiene acceso a esta información, tiene que pasarlo a BASAL.
  2. Verificar si existen dos identificadores activos en ARCONTE asociados al mismo DNI o número de colegiado. En ARCONTE, si es en Adriano, debe pasarlo a BASAL.

2.3. Apertura de tickets

  1. Abrir incidencia REMEDY
  2. Adjuntar datos: nombre, apellidos, DNI, nº colegiado, colegio actual, colegio anterior.
  3. Enviar referencia al solicitante.

2.4. Escalado por errores en códigos externos

Si el error afecta al código del colegio, distribución o códigos externos: Asignar a Basal.

4. Comunicación final al solicitante por correo electrónico desde “Informar usuario”
