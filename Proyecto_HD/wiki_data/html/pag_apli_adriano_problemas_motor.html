Desbloquear procedimiento acumulación (Caso 6.1)

Identificación del caso:

1. Se mira en el arbolito NIGS (imagen arriba izquierda) > Debe haber un nig que contiene a otro, se pincha en el nig contenido.
2. Se mira en Timeline (imagen margen derecho) que ponga terminador por Acumulación.





1. Lo primero es instar al usuario a que ejecute la operación DACU desde la aplicación y en el expediente acumulante, sólo en el caso que no pueda hacerlo - indicando la causa-, procederemos a atender la petición.


2. Comprobamos el NIG del expediente que debamos desbloquear y hacemos el siguiente select:

SELECT * FROM CORE_OWNER.ASUNTOS WHERE NIG=’nig del expte’ ORDER BY ID DESC;


3. De los diferentes registros que pueden aparecer, comprobamos, dentro del campo PROCEDIMIENTO_NUMERO y PROCEDIMIENTO_TIPO, la información que tengamos que modificar; como referencia, la información de PROCEDIMIENTO_TIPO se obtiene del ID que corresponda en la tabla TM_PROCEDIMIENTOS. Por ejemplo, si en procedimiento tipo de la tabla CORE_OWNER.ASUNTOS aparece 13 y queremos conocer a qué tipo de procedimiento hacer referencia, tendremos que usar la siguiente select:

SELECT * FROM CORE_OWNER.TM_PROCEDIMIENTOS WHERE ID=13;


4. Una vez localizado el registro a modificar, ejecutamos el siguiente select para comprobar si el expediente realmente contiene una operación ACU o ACUP

SELECT * FROM CORE_OWNER.ACO_OPE WHERE ASUNTO = id del asunto AND ID_OPERACION_TIPO IN (16,19) ORDER BY ID DESC;

- Si aparece traza en ACO_OPE, copiamos la información del campo ID_TRAM_BPM - Con ese dato copiado hacemos el siguiente select:

SELECT * FROM CORE_OWNER.INSTANCIA_ACTIVA WHERE CASE_INSTANCE_ID = ‘valor de id_tram_bpm’; - Realizamos el siguiente update: - UPDATE CORE_OWNER.INSTANCIA_ACTIVA SET ID_ESTADO=1, FECHA_INICIO= null WHERE ID = xxxxxxxxx;


5. Para localizar el registro correspondiente en la tabla CORE_OWNER.ASUNTOS_HISTORA, hacemos el siguiente select:

SELECT * FROM CORE_OWNER.ASUNTOS_HISTORIA WHERE ASUNTO IN (id de la tabla asuntos) ORDER BY ID DESC;


6. Comprobamos en la tabla CORE_OWNER.ASUNTOS_HISTORIA el valor del campo ID_ESTADO -que debería estar en 6- correspondiente al idmax(id más alto) del procedimiento afectado.


7. Realizamos el siguiente update:

UPDATE CORE_OWNER.ASUNTOS_HISTORIA SET ID_ESTADO=1 WHERE ID IN (xxxxxxxx);


8. Realizamos un update similar pero en este caso en la tabla CORE_OWNER.ASUNTOS.

UPDATE CORE_OWNER.ASUNTOS SET ID_ESTADO=1 WHERE ID IN (XXXXXXXXX);


9. Realizamos la siguiente select.

SELECT * FROM CORE_OWNER.ACUMULAR WHERE ASU_ACUADO = id de la tabla asuntos; - Si se muestra registro, realizamos el siguiente update: update core_owner.acumular set ind_estado='R' WHERE ID IN (xxxxxxxx);


10. Para localizar el registro correspondiente en la tabla VISOR.CONTENT, hacemos el siguiente script:

SELECT * FROM VISOR.CONTENT WHERE UUID LIKE (nig obtenido de la select en asuntos%') AND TYPE = 'binder' order by id desc;


11. Lo siguiente es hacer el siguiente update, sustituyendo únicamente el valor de campo STATE en el registro ‘binder’ correspondiente al expediente afectado.

UPDATE VISOR.CONTENT SET STATE = 'I-001' WHERE ID = xxxxxxxxx


12. Entrar en el asunto desde la aplicación y comprobar que el desbloqueo se ha realizado correctamente. Verificar también el estado en la ficha Documentos.

Desbloquear procedimiento itineración (Caso 6.3)

Identificación del caso:

Se mira en Timeline, debe poner Estado: “Terminado por Inhibición”.





1.- Lo primero es instar al usuario a que solicite el rechazo del expediente al órgano al que se inhibió, se puede conocer la localización del expediente desde la propia aplicación, sólo en el caso que no pueda hacerlo - indicando la causa-, procederemos a atender la petición.


2.- Comprobamos el NIG del expediente que debamos desbloquear y hacemos el siguiente select:

SELECT * FROM CORE_OWNER.ASUNTOS WHERE NIG=’nig del expte’ ORDER BY ID DESC;


3.- De los diferentes registros que pueden aparecer, tenemos que seleccionar el correcto a modificar, pero en este caso debemos guiarnos por el NRG del órgano judicial, que debemos comprobar en la aplicación, y contrastar dicho numero en la tabla CORE-OWNER.ASUNTOS.


4.- Para localizar el registro correspondiente en la tabla CORE_OWNER.ASUNTOS_ITINERACION, hacemos el siguiente select:

SELECT * FROM CORE_OWNER.ASUNTOS_ITINERACION WHERE ASUNTO = ID de la tabla asuntos ORDER BY ID DESC;


5.- Comprobamos en el registro idmax (id más alto), si el campo de la fecha de entrada del órgano actualmente propietario está indicada o está a null.

Caso a. Está indicada: Significa que el órgano itinerado o propietario lo ha aceptado.
Caso b. Está a null: Significa que el órgano itinerado o propietario lo tiene pte de aceptar.


CASO A: el órgano itinerado o propietario lo ha aceptado.


6.- Realizamos el siguiente insert para reenviar el expediente al órgano remitente o reclamante, los valores ‘nº registro’ y ‘clase’ se pueden obtener de registros previos de ese mismo asunto en dicha tabla.

UPDATE CORE_OWNER.ASUNTOS_ITINERACION SET FECHA_SALIDA=fecha de salida, ID_ORGANO_DESTINO=órgano reclamante, FECHA_ENVIO=FECHA_SALIDA, FECHA_RECEPCION=FECHA_SALIDA WHERE ID = idmax;

Desbloquear procedimiento transformación (Caso 6.4)

Identificación del caso:

Se mira en Timeline, debe poner último trámite: TRPROC – Estado: Terminado por transformación.





1.- IMPORTANTE: Los expedientes transformados comparten el mismo idasunto, por lo que no es posible mantener los dos expedientes activos a la vez, por el consiguiente riesgo de errores en la tramitación e inconsistencia de datos, por eso lo primero es preguntar al usuario, caso de que no se especifique en la petición, que especifique si quiere:

a) Desbloquear el expediente original y anular el transformado.
b) Desbloquear sólo el expediente original y bloquear el transformado.


2.- Comprobamos el NIG del expediente que debamos desbloquear y hacemos el siguiente select:

SELECT * FROM CORE_OWNER.ASUNTOS WHERE NIG=’nig del expte’ ORDER BY ID DESC;


3.- De los diferentes registros que pueden aparecer, comprobamos, dentro del campo PROCEDIMIENTO_NUMERO y PROCEDIMIENTO_TIPO, la información relacionada con el procedimiento transformado; como referencia, la información de PROCEDIMIENTO_TIPO se obtiene del ID que corresponda en la tabla TM_PROCEDIMIENTOS. Por ejemplo, si en procedimiento tipo de la tabla CORE_OWNER.ASUNTOS aparece 13 y queremos conocer a qué tipo de procedimiento hace referencia, tendremos que usar la siguiente select:

SELECT * FROM CORE_OWNER.TM_PROCEDIMIENTOS WHERE ID=13;


4.- Una vez localizado el registro a modificar, ejecutamos el siguiente select para comprobar si el expediente realmente contiene una operación TRPROC SELECT * FROM CORE_OWNER.ACO_OPE WHERE ASUNTO = id del asunto AND ID_OPERACION_TIPO = 9 ORDER BY ID DESC;

- Si aparece traza en ACO_OPE, copiamos la información del campo ID_TRAM_BPM

- Con ese dato copiado hacemos el siguiente select:
- SELECT * FROM CORE_OWNER.INSTANCIA_ACTIVA WHERE CASE_INSTANCE_ID = ‘valor de id_tram_bpm’;

- Realizamos el siguiente update:
- UPDATE CORE_OWNER.INSTANCIA_ACTIVA SET ID_ESTADO=1, FECHA_INICIO= null WHERE ID = xxxxxxxxx;


5.- Para localizar el registro correspondiente en la tabla CORE_OWNER.ASUNTOS_HISTORA, hacemos el siguiente select:

SELECT * FROM CORE_OWNER.ASUNTOS_HISTORIA WHERE ASUNTO IN (id de la tabla asuntos) ORDER BY ID DESC;


6.- Comprobamos en la tabla CORE_OWNER.ASUNTOS_HISTORIA que, asociado al mismo número de asunto y mismo órgano, aparecen registros con tipo y número de procedimiento diferentes; al tener ordenados los registros en orden descendente de id podremos comprobar que los registros superiores están asociados al tipo y numero de procedimiento transformado mientras que los inferiores corresponden al tipo y numero de procedimiento original.




CASO A: Desbloquear el expediente original y anular el transformado.


7. Realizamos el siguiente update en el id máximo/superior correspondiente al expediente transformado.

UPDATE CORE_OWNER.ASUNTOS_HISTORIA SET ES_INCOACION_NUEVO=0, PROCEDIMIENTO_TIPO= código del procedimiento original ,PROCEDIMIENTO_NUMERO=numero del procedimiento original, ID_ESTADO=1, ID_PADRE=id previo al id máximo WHERE ID = xxxxxxxxx;


8.- Realizamos otro update también en la tabla CORE_OWNER.ASUNTOS_HISTORIA pero esta vez en el id superior pero correspondiente al expediente original.

UPDATE CORE_OWNER.ASUNTOS_HISTORIA SET ACTIVO=0 WHERE ID = xxxxxxx;


9.- Realizamos el siguiente update pero en este caso en la tabla CORE_OWNER.ASUNTOS.

UPDATE CORE_OWNER.ASUNTOS SET PROCEDIMIENTO_TIPO=código del procedimiento original, PROCEDIMIENTO_NUMERO=número del procedimiento original, ID_ESTADO=1 WHERE ID = xxxxxxxx;


10.- Lo siguiente será liberar el número de procedimiento transformado que hemos de anular, en ese caso tenemos que localizar el numerador correspondiente en la tabla CORE_OWNER.NUMERADORES, para ello tenemos que localizar, en la primera select que hicimos en el punto 1, el código id del órgano judicial; respecto al numerador correspondiente lo debemos obtener de la tabla TM_CORE_OWNER_NUMERADORES_TIPO en conjugación con la tabla TM_PROCEDIMIENTOS.

Hacemos la siguiente select:

SELECT * FROM CORE_OWNER.NUMERADORES where x_organismo = id del órgano AND ANYO=2024 AND ID_TIPO_NUMERADOR IN (id del numerador);


11.- Lo siguiente es hacer el siguiente insert:

Insert into CORE_OWNER.NUMERADORES (ID,NUMERO,PROXIMO_O_LIBRE,ID_TIPO_NUMERADOR,X_ORGANISMO,ANYO,VERSION,ID_JURISDICCION) values (CORE_OWNER.SEQ_NUMERADORES_ID.NEXTVAL,'número sustituido','L','id del órgano',’id del numerador’,'2024',0,jurisdicción que corresponda;


12.- Para localizar el registro correspondiente en la tabla VISOR.CONTENT, hacemos el siguiente script:

SELECT * FROM VISOR.CONTENT WHERE UUID LIKE (nig obtenido de la select en asuntos%') AND TYPE = 'binder' order by id desc;


13.- Lo siguiente es hacer el siguiente Delete, a fin de eliminar del visor el registro correspondiente a la carpeta contenedora (binder) del expediente transformado que hemos anulado.

CASUÍSTICA:

Se pueden plantear principalmente dos casuísticas sobre numeradores de resolución:

1.- Solicitan modificar el numerador dado a una resolución.
2.- Solicitan liberar un número de resolución.

Error numerador resolución-ya asignado-(Caso 1.1)

- En primer lugar, debemos considerar dos aspectos:


a) Que el número de resolución solicitado a añadir o reemplazar no esté ya asociado a otra resolución anterior. En este caso debemos conocer el id del órgano afectado, para ello podemos utilizar la siguiente select a partir del NIG indicado en la incidencia:

SELECT * FROM CORE_OWNER.ASUNTOS WHERE NIG = 'xxxxxxxxxxxxxxxxxxx’;


Debemos comprobar el valor del campo ID_ORGANO con la siguiente tabla:

select * from core_owner.organos where id =id_organo de la select anterior;


Una vez verificado que el código del órgano es el correcto, procedemos con la siguiente select que nos mostrará todas las resoluciones de ese órgano. Las jurisdicciones en @driano se identifican como: 1=Civil; 2=Penal; 3=Cont-Advo; 4=Social.

SELECT * FROM CORE_OWNER.resoluciones where asunto in (select id from core_owner.asuntos where id_organo=xxxxx) and id_resolucion_tipo = (1, 2 o 3) and JURISDICCION=(1,2,3 o 4) AND resolucion_numero is not null order by resolucion_numero desc;


Otra opción sería:

SELECT * FROM CORE_OWNER.resoluciones where asunto in (select id from core_owner.asuntos where id_organo=xxxxx) and id_resolucion_tipo=(1,2 o 3) and resolucion_numero=nº resolución que quieren añadir/modificar;


b) Que la resolución a modificar no esté ya notificada.

Una manera de comprobar si la resolución ha sido notificada es localizar el id del documento de la resolución:

SELECT * FROM CORE_OWNER.DOCUMENTOS WHERE ASUNTO=ID DEL ASUNTO;


Se nos mostrará la relación de documentos del expediente, tenemos que localizar la resolución (Auto, Sentencia o Decreto); para ello podemos comprobar campos de referencia como NOM_ADJUNTO, DESC_ADJUNTO, DESC_NOM, y también CLASE que nos indicará si el documento tiene asociada correctamente la clase correspondiente (SENT, AUTR, DECR). Una vez identificado el documento, copiamos su ID y ejecutamos la siguiente select:

SELECT * FROM CORE_OWNER.notifics_lxn_docs WHERE ASUNTO=id del asunto AND ID_DOCUMENTO = xxxxxxxxx;


Si esta select muestra traza, será indicativo de que la resolución ha sido notificada y, si copiamos el ID_NOTIFICACION, podemos obtener el detalle de la notificación con la siguiente select:

SELECT * FROM CORE_OWNER.NOTIFICS_LXN WHERE id= id_notificacion de la select anterior;


- Hechas las comprobaciones anteriores y, en caso de que no se cumpla ninguna de ellas, procedemos a modificar el número de la resolución solicitado.


- Lo primero de todo será localizar el expediente asociado a la resolución, para ello contamos con el NIG del expediente que debe indicarse en la incidencia y ejecutamos el select anterior sonbre la tabla ASUNTOS:

SELECT * FROM CORE_OWNER.ASUNTOS WHERE NIG = 'xxxxxxxxxxxxxxxxxxx’;


-El siguiente paso será copiar el id del asunto y realizar la siguiente select que nos permitirá conocer el número actual de la resolución que quieren modificar.

SELECT * FROM CORE_OWNER.resoluciones where asunto =id_asunto and id_resolución_tipo=(1,2 o 3)


- Copiamos el ID de la resolución.


-Por último, procedemos a modificar el número actual de la resolución por el que nos indiquen.

Liberar un número de resolución

- Se trata del supuesto en el que, como consecuencia de un salto imprevisto de numerador, se ha producido una quiebra de la numeración secuencial de resoluciones. Liberar el número saltado supone reparar esa quiebra secuencial mediante el relleno de número omitido y que el usuario normalmente suele indicar, en caso de no indicarlo debemos localizarlo a través de la select sobre resoluciones indicada en el punto 2.1.a

- Se pueden generar tantos numeradores libres como sea necesario ya que suelen ser frecuentes las peticiones de liberación de varios números para un mismo juzgado.

- La tabla afectada será la de NUMERADORES, de manera que el insert a utilizar requerirá previamente la identificación del órgano judicial y del tipo del tipo de resolución sobre la que vayamos a trabajar.

- La identificación del órgano se realiza tal y como ya se ha explicado en el punto 2.1.a anterior.

- La identificación del numerador para la resolución viene indicado en el punto 1 anterior.

IMPORTANTE: Tener en cuenta que los juzgados mixtos (Civil-Penal) tienen el mismo numerador de resolución para ambas jurisdicciones, por ello a la hora de liberar un número es importante tener en cuenta la jurisdicción sobre la que estamos aplicando (1=Civil - 2=Penal).



- El insert a aplicar será:

Insert into CORE_OWNER.NUMERADORES (ID,NUMERO,PROXIMO_O_LIBRE,ID_TIPO_NUMERADOR,X_ORGANISMO,ANYO,VERSION,ID_JURISDICCION) values (CORE_OWNER.SEQ_NUMERADORES_ID.NEXTVAL,'0000xxx','L','tipo_numerador','id del organo','año','0',’ID_JURISDICCION’);
